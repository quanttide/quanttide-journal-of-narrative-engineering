# 2026-06-03

## Provider API 完全重写：StyleSample 驱动的 3R

今日完成 Provider API 的第四次重写——从"段落级起承转合分析"演进为"StyleSample 驱动的多维度风格对齐评估"。

### 架构演进史

| 阶段 | 核心模型 | 问题 |
|------|---------|------|
| alpha.1 | 段落级起承转合 + StyleStore | 有状态副作用，LLM stub |
| alpha.2 | 无状态 /review + /reflect + /rewrite + /cycle | 双管线（Flutter 正则 ≠ Provider LLM） |
| alpha.3 | 无状态 /review + 3R 端点 | summary 硬编码，style_examples 硬编码 |
| alpha.4 | Criterion 驱动（positive/negative/constraint） | criteria 模型太扁平，缺乏维度结构 |
| **alpha.5** | **StyleSample 驱动（dimensions + excerpts）** | — |

### 最终 API

| 端点 | 输入 | 输出 |
|------|------|------|
| `POST /review` | `text` + `style{dimensions, excerpts}` | `dimension_alignments[]`（每个维度的对齐分数 + 偏差） |
| `POST /analyze` | `text` + `style` + `dimension_title` | 深度分析：pattern 对比 + root_cause + fix_strategies |
| `POST /inspire` | `text` + `style` + `target_dimensions[]` + `variety` | 多条启发式改写建议 |

### 核心设计决策

**1. Style 是一等公民**

放弃"段落评审"和"通用质量标准"的概念。系统不再内置任何写作规则——用户通过 `Style` 定义"什么算好"。`Style` 包含：

| 字段 | 来源 | 用途 |
|------|------|------|
| `dimensions[]` | `style.yaml` | LLM 的分析框架（叙事视角、语言风格、情感表达等） |
| `excerpts[]` | `style.yaml` | 每个维度的短示例，嵌入 prompt |
| `samples[]` | `sample.md` | 完整样本（预留，暂未接入 prompt） |

**2. 三个端点 ≠ 三个独立工具**

`/review` 做广度（所有维度的概览），`/analyze` 做深度（单个维度的根因），`/inspire` 做创意（基于偏差生成建议）。前端工作流：review → 低分维度 → analyze → 看根因 → inspire → 获取改法。

**3. 集成测试只验证稳定的东西**

14 个集成测试的经验教训：LLM 输出不可预测，唯一可靠的断言是存在性（字段有值）和相对比较（同一请求内 A > B）。试图对 LLM 输出设数值阈值是在和随机性赌博。
